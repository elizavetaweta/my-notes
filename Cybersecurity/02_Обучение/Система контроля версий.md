#tech #tool 

**Система контроля версий** — это инструмент, который отслеживает и фиксирует каждое изменение в файле. Позволяет восстановить предыдущие версии, сравнить их, и выяснить, кто и когда внёс правку.

Для специалиста по ИБ это не просто «инструмент разработчика», а **средство обеспечения целостности** и **обязательный элемент аудита изменений**.


### 🛡 Роль в ИБ:
- **История изменений:** Каждое действие записывается (кто, когда и что изменил).
- **Защита от потери данных:** Если файл поврежден или удален, его можно восстановить из истории (Snapshot).
- **Безопасная разработка:** Позволяет проверять код (Code Review) перед тем, как он попадет в рабочую систему.

## Ключевые понятия

| Термин | Значение |
|---|---|
| **Репозиторий (repo)** | Хранилище файлов и всей истории изменений |
| **Коммит (commit)** | Зафиксированный снимок состояния файлов с уникальным хешем и сообщением |
| **Ветка (branch)** | Изолированная линия разработки, позволяющая вести работу параллельно, не затрагивая основную версию |
| **Слияние (merge)** | Объединение изменений из одной ветки в другую |
| **Pull request / Merge request** | Запрос на влитие изменений с обязательным ревью |
| **Тег (tag)** | Метка, фиксирующая важную точку (например, релиз версии) |

## Распространённые системы контроля версий

- **Git** — де-факто стандарт. Распределённая система, используется на GitHub, GitLab, Bitbucket. Повсеместно применяется в ИБ, в том числе для хранения IaC-конфигураций и документации.
- **Subversion (SVN)** — централизованная система контроля версий старшего поколения, ещё встречается в крупных enterprise-средах.
- **Mercurial** — распределённая, схожа с Git, но менее популярна.

### ⚙️ Как работает проверка целостности в Git:
Git использует **[[Хеширование]]** (алгоритм SHA-1) для идентификации каждого изменения. Если данные внутри файла изменятся без ведома системы, хеш не совпадет, и Git сообщит об ошибке.

## Пример из практики ИБ

> Конфигурация корпоративного межсетевого экрана хранится в Git-репозитории. Любое изменение правил фильтрации проходит через pull request с обязательным ревью со стороны второго инженера ИБ. После слияния автоматически запускается проверка целостности конфигурации. При инциденте по логам коммитов видно, кто и когда внёс конкретное правило, и можно мгновенно откатиться до предыдущей безопасной версии.

Этот же подход применяется к скриптам автоматизации, политикам SELinux/AppArmor, конфигурациям SIEM и другим компонентам безопасности.

###  Как система контроля версий помогает при расследовании инцидента?

При инциденте система контроля версий позволяет ответить на ключевые вопросы:
- **Когда** именно произошло внедрение уязвимости или вредоносного изменения? (просмотр лога коммитов по времени)
- **Кто** внёс изменение? (автор коммита, возможно с GPG-подписью)
- **Что именно** было изменено? (сравнение версий, `git diff`)
- **Было ли изменение легитимным?** (проверка наличия pull request, аппрувов)

Таким образом, система контроля версий даёт возможность **восстановить хронологию атаки**, определить точку входа и оценить масштаб нарушения. Это критически важно для последующего закрытия уязвимости и корректировки процессов.


### Чем коммит принципиально отличается от простого сохранения файла?

Простое сохранение (Ctrl+S) перезаписывает файл, теряя предыдущее состояние. **Коммит** же:
- создаёт **снимок состояния всех файлов** на определённый момент;
- привязывает к нему **автора, время и сообщение** с описанием изменений;
- вычисляет **уникальный хеш**, который делает невозможным незаметное изменение истории;
- сохраняет **всю цепочку предшествующих коммитов**, обеспечивая полную прослеживаемость.

Таким образом, коммит — это атомарная, подписанная и проверяемая точка в истории проекта, а не просто новые данные на диске.
