# 🔧 OpenSSL — практическое руководство для специалиста по ИБ

> [!abstract] Суть одной фразой
> OpenSSL — это криптографический инструментарий с открытым исходным кодом, реализующий протоколы [[TLS]] и [[SSL]], а также предоставляющий доступ к обширному набору криптографических функций из командной строки. Для специалиста по ИБ это «швейцарский нож» для работы с сертификатами, ключами, шифрованием и диагностикой защищённых соединений.

OpenSSL выполняет три главные функции для ИБ-специалиста:
1.  **Управление ключами и сертификатами:** создание [[RSA]], [[ECC]] ключей, запросов на подпись (CSR), самоподписанных сертификатов [[X.509]].
2.  **Криптографические операции:** [[Шифрование]], [[Хеширование]], создание и проверка [[Электронная подпись|электронной подписи]].
3.  **Диагностика TLS/SSL:** тестирование защищённых соединений, просмотр сертификатов серверов, проверка цепочек доверия [[Инфраструктура открытых ключей (PKI)]].

Знание этих команд необходимо для аттестации, пентеста и повседневного администрирования.

## 1. Работа с ключами
### 🔑 Генерация закрытого ключа RSA
> <span style="color:rgb(146, 208, 80)">openssl genrsa -out private.key 2048</span>
**Что делает:** генерирует закрытый RSA-ключ длиной 2048 бит и сохраняет его в файл `private.key`.
**Разбор опций:**
- `genrsa` — подкоманда генерации RSA-ключа.
- `-out private.key` — файл, в который будет записан ключ.
- `2048` — длина ключа в битах. Минимально допустимый уровень для современных систем. Для долговременной защиты или гостайны рекомендуется 4096 бит.
**Когда используется:** при создании нового сервера, генерации ключа для SSL/TLS-сертификата, создании ключевой пары для аутентификации.

> <span style="color:rgb(146, 208, 80)">openssl genrsa -aes256 -out private_encrypted.key 4096</span>
**Что делает:** генерирует RSA-ключ длиной 4096 бит и **защищает его паролем** с помощью симметричного шифрования AES-256.
**Разбор опций:**
- `-aes256` — шифровать ключ на диске алгоритмом [[AES]] с 256-битным ключом. При использовании ключа система запросит пароль.
- `4096` — увеличенная длина ключа для высокой стойкости.
**Когда используется:** когда закрытый ключ должен храниться на диске или в системе, к которой возможен доступ посторонних. Без этой опции ключ лежит в открытом виде, что критично при утечке файла.
### 🔑 Генерация ключа ECC
> <span style="color:rgb(146, 208, 80)">openssl ecparam -genkey -name prime256v1 -out private_ecc.key</span>
**Что делает:** генерирует закрытый ключ на эллиптической кривой `prime256v1` (она же `secp256r1` или `P-256`).
**Разбор опций:**
- `ecparam` — работа с параметрами эллиптической криптографии ([[ECC]]).
- `-genkey` — указание, что нужно сгенерировать ключ.
- `-name prime256v1` — имя стандартизированной кривой. `prime256v1` — наиболее распространённая кривая с уровнем стойкости ~128 бит (аналог RSA-3072).
- `-out private_ecc.key` — файл для сохранения ключа.
**Когда используется:** для современных систем, где требуется компактный и быстрый ключ. ECC предпочтительнее RSA для новых проектов: сертификаты, TLS, IoT.
### 🔑 Извлечение открытого ключа
> <span style="color:rgb(146, 208, 80)">openssl rsa -in private.key -pubout -out public.key</span>
**Что делает:** извлекает открытый ключ из закрытого RSA-ключа.
**Разбор опций:**
- `rsa` — подкоманда работы с RSA-ключами.
- `-in private.key` — входной файл с закрытым ключом.
- `-pubout` — вывести **только открытую часть** ключа.
- `-out public.key` — сохранить открытый ключ в файл.

> <span style="color:rgb(146, 208, 80)">openssl ec -in private<i>ecc.key -pubout -out public</i>ecc.key</span>
**Что делает:** то же самое для ECC-ключа.
- `ec` — подкоманда работы с эллиптическими ключами.
**Когда используется:** открытый ключ нужно передать коллегам, администратору сервера, в удостоверяющий центр для выпуска сертификата, или опубликовать.

## 2. Работа с сертификатами X.509 и PKI
### 📜 Создание самоподписанного сертификата
> <span style="color:rgb(146, 208, 80)">openssl req -x509 -newkey rsa:4096 -keyout ca<i>key.pem -out ca</i>cert.pem -days 365 -nodes</span>
**Что делает:** создаёт самоподписанный сертификат и ключ для него за одну команду. Часто используется для создания корневого сертификата УЦ (CA) для тестового стенда или внутреннего использования.
**Разбор опций:**
- `req` — подкоманда работы с запросами на сертификат (CSR) и самоподписанными сертификатами.
- `-x509` — указывает, что нужно создать самоподписанный сертификат, а не запрос на подпись.
- `-newkey rsa:4096` — создать новый RSA-ключ длиной 4096 бит.
- `-keyout ca_key.pem` — сохранить ключ в этот файл.
- `-out ca_cert.pem` — сохранить сертификат в этот файл.
- `-days 365` — срок действия сертификата в днях.
- `-nodes` — **No DES**, то есть не шифровать закрытый ключ паролем. Для корневого УЦ в продакшене так делать **нельзя** — ключ должен быть защищён.
### 📜 Создание сертификата, подписанного вашим УЦ (CA)
**Шаг 1 — создать запрос (CSR):**
> <span style="color:rgb(146, 208, 80)">openssl req -new -newkey rsa:2048 -nodes -keyout server<i>key.pem -out server</i>req.csr</span>
**Что делает:** создаёт новый ключ и **запрос на подпись сертификата** (Certificate Signing Request).
**Разбор опций:**
- `req -new` — создать новый CSR.
- `-newkey rsa:2048` — создать новый RSA-ключ длиной 2048 бит.
- `-nodes` — не шифровать ключ паролем.
- `-keyout server_key.pem` — файл для закрытого ключа сервера.
- `-out server_req.csr` — файл с запросом, который будет отправлен в УЦ.
**Когда используется:** когда нужно получить сертификат для веб-сервера от внешнего или внутреннего УЦ.
**Шаг 2 — подписать запрос:**
> <span style="color:rgb(146, 208, 80)">openssl x509 -req -in server<i>req.csr -CA ca</i>cert.pem -CAkey ca<i>key.pem -CAcreateserial -out server</i>cert.pem -days 365</span>
**Что делает:** подписывает CSR с помощью корневого сертификата и его ключа, создавая готовый сертификат.
**Разбор опций:**
- `x509` — работа с сертификатами X.509.
- `-req` — на входе запрос на подпись (CSR), а не сертификат.
- `-in server_req.csr` — входной CSR-файл.
- `-CA ca_cert.pem` — сертификат УЦ, которым будет подписан запрос.
- `-CAkey ca_key.pem` — закрытый ключ этого УЦ.
- `-CAcreateserial` — создать серийный номер для нового сертификата (если файл serial отсутствует).
- `-out server_cert.pem` — готовый сертификат.
- `-days 365` — срок действия.
**Когда используется:** при развёртывании собственного центра сертификации внутри организации.
### 📜 Проверка цепочки сертификатов
> <span style="color:rgb(146, 208, 80)">openssl verify -CAfile ca<i>cert.pem server</i>cert.pem</span>
**Что делает:** проверяет, что сертификат `server_cert.pem` подписан УЦ, чей сертификат находится в `ca_cert.pem`, и что цепочка доверия валидна.
**Разбор опций:**
- `verify` — подкоманда проверки сертификатов.
- `-CAfile ca_cert.pem` — файл с доверенным корневым сертификатом.
**Когда используется:** при аудите и настройке веб-сервера — убедиться, что сертификат не поддельный и цепочка не нарушена.

## 3. Диагностика TLS/SSL-соединений
### 🌐 Подключение к серверу и просмотр его сертификата
> <span style="color:rgb(146, 208, 80)">openssl s<i>client -connect example.com:443 -servername example.com -tls1</i>2</span>
**Что делает:** устанавливает TLS-соединение с сервером и выводит на экран всю информацию о сертификате, цепочке и параметрах соединения.
**Разбор опций:**
- `s_client` — SSL/TLS-клиент для диагностики.
- `-connect example.com:443` — адрес и порт сервера.
- `-servername example.com` — необходимо для корректной работы SNI (Server Name Indication), чтобы сервер вернул правильный сертификат, если на одном IP несколько сайтов.
- `-tls1_2` — принудительно использовать TLS 1.2 (можно заменить на `-tls1_3` или убрать для согласования наивысшей версии).
**Когда используется:** при пентесте, аудите веб-приложений, диагностике проблем с сертификатами.
> openssl s_client -connect example.com:443 -showcerts
**Что делает:** то же самое, но показывает все сертификаты в цепочке доверия (не только конечный).
**Когда используется:** для анализа всей цепочки — нужно убедиться, что промежуточные сертификаты присутствуют и валидны.
### 🌐 Проверка поддержки конкретных протоколов
> <span style="color:rgb(146, 208, 80)">openssl s<i>client -connect example.com:443 -tls1</i>3</span>
> <span style="color:rgb(146, 208, 80)">openssl s_client -connect example.com:443 -ssl3</span>
**Что делает:** проверяет, поддерживает ли сервер указанную версию протокола. Если соединение успешно — поддерживает. Если ошибка (`no peer certificate available`) — не поддерживает, что для SSL3 является нормой и требованием безопасности.
**Когда используется:** при аудите для выявления старых, уязвимых протоколов (SSL 2.0, SSL 3.0, TLS 1.0).
### 🌐 Проверка срока действия сертификата
> <span style="color:rgb(146, 208, 80)">echo | openssl s_client -servername example.com -connect example.com:443 2>/dev/null | openssl x509 -noout -dates</span>
**Что делает:** подключается к серверу, забирает его сертификат и выводит только даты начала и окончания срока действия.
**Разбор опций:**
- `echo |` — автоматически закрывает соединение после получения сертификата, не дожидаясь ввода.
- `2>/dev/null` — скрывает stderr, чтобы вывод был чистым.
- `| openssl x509 -noout -dates` — передаёт полученный сертификат в подкоманду `x509`, которая извлекает только поля `notBefore` и `notAfter`.
**Когда используется:** в скриптах мониторинга и при быстром аудите.

## 4. Криптографические операции
### ✂️ Хеширование
> <span style="color:rgb(146, 208, 80)">openssl dgst -sha256 file.txt</span>
**Что делает:** вычисляет хеш-сумму файла по алгоритму [[SHA‑2 (SHA‑256 и SHA‑512)|SHA‑256]].
**Разбор опций:**
- `dgst` — подкоманда вычисления дайджеста (хеша).
- `-sha256` — использовать SHA‑256. Можно заменить на `-sha512` или `-md5`.
> echo -n "Hello World" | openssl dgst -sha256
**Что делает:** вычисляет хеш от переданной строки (без символа перевода строки из-за `-n`).
**Когда используется:** проверка целостности скачанных файлов, создание контрольных сумм, сравнение хешей.
### 🔐 Симметричное шифрование
> <span style="color:rgb(146, 208, 80)">openssl enc -aes-256-cbc -pbkdf2 -in secret.txt -out secret.enc</span>
**Что делает:** шифрует файл алгоритмом [[AES]]-256 в режиме [[Режимы блочного шифрования|CBC]].
**Разбор опций:**
- `enc` — подкоманда шифрования/расшифрования.
- `-aes-256-cbc` — алгоритм и режим.
- `-pbkdf2` — использовать современный алгоритм формирования ключа из пароля (Password-Based Key Derivation Function 2). **Критически важно** для безопасности: без этой опции OpenSSL использует слабый алгоритм.
- `-in secret.txt` — входной файл.
- `-out secret.enc` — выходной (зашифрованный) файл.
> openssl enc -d -aes-256-cbc -pbkdf2 -in secret.enc -out decrypted.txt
**Что делает:** расшифровывает файл обратно. `-d` означает decrypt.
**Когда используется:** защита файлов на диске, передача зашифрованных данных, резервное копирование.
### 🔐 Асимметричное шифрование
> <span style="color:rgb(146, 208, 80)">openssl pkeyutl -encrypt -in plain.txt -pubin -inkey public.key -out encrypted.bin</span>
**Что делает:** шифрует небольшой файл открытым ключом.
**Разбор опций:**
- `pkeyutl` — подкоманда для низкоуровневых операций с асимметричной криптографией.
- `-encrypt` — режим шифрования.
- `-pubin` — указание, что на входе **открытый** ключ.
- `-inkey public.key` — файл с ключом.
- `-out encrypted.bin` — зашифрованный выходной файл.
> openssl pkeyutl -decrypt -in encrypted.bin -inkey private.key -out decrypted.txt
**Что делает:** расшифровывает файл закрытым ключом.
**Когда используется:** для безопасной передачи симметричного сессионного ключа или другого небольшого секрета. Не применяется для больших файлов.
### 🖊️ Электронная подпись
> <span style="color:rgb(146, 208, 80)">openssl dgst -sha256 -sign private.key -out signature.bin document.txt</span>
**Что делает:** создаёт отсоединённую электронную подпись файла с помощью закрытого ключа.
**Разбор опций:**
- `dgst` — хеширование с возможностью подписи.
- `-sha256` — сначала вычисляется хеш файла, затем подписывается.
- `-sign private.key` — подписать закрытым ключом.
- `-out signature.bin` — файл с подписью.
> openssl dgst -sha256 -verify public.key -signature signature.bin document.txt
**Что делает:** проверяет подпись с помощью открытого ключа. Выводит `Verified OK` или `Verification Failure`.
**Когда используется:** проверка подлинности и целостности документов, файлов конфигурации, скриптов.

## 🧠 Значимость для меня как специалиста по ИБ

- Я использую OpenSSL как основной инструмент для создания и анализа сертификатов, генерации ключей и проведения криптографических операций.
- При аудите и пентесте я проверяю поддержку протоколов и шифров, валидность сертификатов и их цепочек с помощью команд `s_client` и `verify`.
- При настройке серверов я умею быстро сгенерировать самоподписанный сертификат для тестового стенда или сформировать корректный CSR для отправки в центр сертификации.
- Я всегда помню о безопасности: не передаю пароли к ключам в открытом виде, использую `-pbkdf2` для шифрования файлов и храню закрытые ключи в недоступном для посторонних месте.

## ❓ Самопроверка

1.  **Как сгенерировать ECC-ключ?**
    👉 `openssl ecparam -genkey -name prime256v1 -out private.key`

2.  **Как проверить цепочку сертификата?**
    👉 `openssl verify -CAfile root.pem cert.pem`

3.  **Как быстро проверить срок действия сертификата на сервере?**
    👉 `echo | openssl s_client -servername example.com -connect example.com:443 2>/dev/null | openssl x509 -noout -dates`

4.  **В чём отличие RSA от ECC и какой выбрать?**
    👉 ECC предпочтительнее: он быстрее и требует меньше ресурсов при том же уровне стойкости (ключ 256 бит ECC ≈ 3072 бит RSA).
