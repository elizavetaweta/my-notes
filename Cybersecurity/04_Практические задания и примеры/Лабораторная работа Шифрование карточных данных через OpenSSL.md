# 🔐 Лабораторная работа: Шифрование карточных данных через OpenSSL

## 🎯 Суть задания (Что мы делали)
Мы смоделировали процесс безопасной передачи данных банковской карты с мобильного устройства на сервер. Чтобы данные не передавались в открытом виде, мы:
1. Привели их к строгому текстовому шаблону.
2. Зашифровали асимметричным алгоритмом **RSA**.
3. Закодировали в формат **Base64** для безопасной передачи в виде текстовой строки через API.

Для проверки мы выполнили обратный процесс (декодирование + дешифрование) и успешно восстановили исходный текст.

---

## 🛠 Часть 1: Процесс шифрования

### Шаг 1. Форматирование данных
```bash
echo -n "PAN=4300000000000777;ExpDate=0823;CVV=999" > message.txt
```
* **`echo`** — выводит указанный текст в поток данных.
* **`>`** — оператор перенаправления. Переносит текст из терминала в файл `message.txt`.
* **`-n`** — **флаг отмены переноса строки**. Сокращение от *no newline* (буква `n` означает *newline*, а дефис `-` выступает как отрицание стандартного поведения). Без этого флага в конец файла запишется скрытый символ `\n`, что полностью изменит итоговый шифр.

### Шаг 2. Асимметричное шифрование (RSA)
```bash
openssl pkeyutl -encrypt -pubin -inkey public.key -in message.txt -out cypher.txt
```
* **`openssl pkeyutl`** — встроенная утилита (Public Key Utility) для работы с асимметричной криптографией. Она автоматически определяет алгоритм (RSA) по структуре файла ключа.
* **`-encrypt`** — включает режим зашифрования.
* **`-pubin`** — сообщает, что для шифрования используется **публичный (открытый) ключ**.
* **`-inkey public.key`** — путь к файлу с публичным ключом.
* **`-in message.txt`** — файл-источник с исходными данными карты.
* **`-out cypher.txt`** — выходной файл, содержащий нечитаемые бинарные данные (сырые байты).

### Шаг 3. Кодирование в Base64
```bash
openssl enc -base64 -A -in cypher.txt -out msg.base64
```
* **`enc -base64`** — переводит бинарный файл в безопасный текстовый формат Base64 (буквы и цифры), который не повредится при передаче по сети.
* **`-A`** — **критический флаг**. Заставляет утилиту упаковать весь текст в **одну сплошную строку** без принудительных переносов (по умолчанию Base64 разбивает текст на строки каждые 64 символа). Данные в одну строку — стандарт для передачи через API.
* **`-out msg.base64`** — итоговый файл. Строка из него является финальным результатом подзадания.

## 🔑 Часть 2: Процесс расшифрования (Проверка)

### Шаг 4. Декодирование из Base64
```bash
openssl enc -d -base64 -A -in msg.base64 -out message.txt
```
* **`-d`** — флаг обратного действия / декодирования (Decrypt). 
* Возвращает текст из формата Base64 в исходный бинарный зашифрованный вид и перезаписывает файл `message.txt`.

### Шаг 5. Дешифрование приватным ключом (RSA)
```bash
openssl pkeyutl -decrypt -inkey private.key -in message.txt -out decrypted.txt
```
* **`-decrypt`** — запускает процесс дешифрования.
* **`-inkey private.key`** — использует секретный приватный ключ сервера. Только он способен расшифровать то, что было закрыто публичным ключом из Шага 2.

### Шаг 6. Просмотр и верификация результата
```bash
cat decrypted.txt
```
* **Вывод терминала:** `PAN=4300000000000777;ExpDate=0823;CVV=999%`
* Символ `%` в конце строки (особенность Zsh в Manjaro) подтверждает, что текст успешно расшифрован и в нем нет лишних скрытых символов переноса строки.

## 🧠 Важная теория для зачёта

> [!info] Шифрование vs Кодирование
> * **Шифрование (RSA)** — защищает данные от чтения конфиденциальным ключом. Без приватного ключа прочесть текст невозможно.
> * **Кодирование (Base64)** — не защищает данные, а лишь меняет их формат (из байтов в текст) для удобства передачи. Ключи не нужны, раскодировать его может любой желающий.

> [!tip] Логика флага `-n` (no newline)
> В Linux утилиты устроены по принципу экономии букв. Поведение `echo` по умолчанию — всегда добавлять перенос строки (*newline*). Буква `n` указывает на эту сущность, а дефис перед ней работает как переключатель: **«отключить стандартный newline»**.
